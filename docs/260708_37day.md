## 2교시: 라이브러리, 프레임워크

- 라이브러리 (Library)
    - 애플리케이션 개발에 필요한 특정 기능들이 미리 구현된 재사용 가능한 코드, 클래스, 함수의 집합
    - 날짜 계산이나 데이터 암호화처럼 복잡하고 반복적인 기능 코드들 매번 직접 설계하지 않고, 이미 품질과 신뢰성이 보장된 리소스를 활용하여 전체 결함을 낮추고 개발 생산성을 높이기 위해 필요함
    - 개발자가 전체적인 프로그램의 실행 흐름을 직접 제어하며, 필요한 시점에 적절한 라이브러리를 주도적으로 호출하며 사용함
- 프레임워크 (Framework)
    - 애플리케이션을 완성하는 데 필요한 전체적인 설계 구조와 구동 규칙을 기본 뼈대로 미리 만들어 제공하고, 개발자가 세부적인 기능 코드를 약속된 위치에 끼워 넣으면 프로그램이 가동될 때 적절한 타이밍에 개발자가 코드를 호출하여 전체 실행 흐름을 통제해 주는 소프트웨어 개발의 기반이 되는 플랫폼
    - 다수의 작업자가 참여하는 거대 프로젝트에서 프로그램의 기본 뼈대와 규칙을 강제함으로써, 개발자 개개인의 설계 차이에서 오는 품질 격차를 최소화하고 코드를 일관되게 정립하기 위해 필요
    - 데이터 접근이나 사용자 인증 등 매번 공동 설계가 필요한 작업은 프레임워크에 위임하여 생산 효율을 높이고, 개발자는 핵심 비즈니스 구현에만 집중할 수 있음
    - 개발자는 프레임워크가 정의한 설계 규칙에 따라 코드를 작성하여 삽입하며, 프레임워크가 이 코드를 적절한 타이밍에 호출하여 실행함
    - **제어의 역전 (IoC, Inversion of Control)**
        - 개발자가 직접 객체를 생성하고 코드의 실행 흐름을 주도하는 것이 아니라, 프레임워크가 객체의 생명주기를 관리하고 작성된 코드를 알맞은 타이밍에 실행하여 흐름을 통제하는 소프트웨어 설계 원칙
        - 스프링 프레임워크에서는 스프링 컨테이너가 객체의 생성 및 소멸, 관계 설정 처리를 대행하여, 개발자가 직접 흐름을 통제하지 않고 핵심 비즈니스 로직 설계에만 전념할 수 있도록 지원함
        - 영화배우 오디션에서 담당자들이 지원자에게 자주 언급하는 헐리우드 원칙(”Don’t call us, we’ll call you”)에서 유래한 소프트웨어 설계 규칙

          [[개체지향 원칙] 헐리우드 원칙(Hollywood Principle)](https://tango1202.github.io/principle/principle-hollywood/)

          [헐리우드 원칙(Hollywood Principle)](https://johngrib.github.io/wiki/jargon/hollywood-principle/)

        - 개발자가 프레임워크의 코드를 직접 호출하여 가동하지 않고, 정해진 규칙에 따라 코드를 등록해 두면 프레임워크가 필요할 때 해당 코드를 호출

## 3교시: 스프링 프레임워크, 빌드, 배포, Maven, Gradle

- 스프링 프레임워크
    - 자바 엔터프라이즈 애플리케이션 개발을 편리하게 해주는 오픈소스 프레임워크
    - 장점
        - 데이터베이스 연결이나 사용자 인증처럼 번거롭고 복잡한 기본 설정 및 인프라 구축 작업을 대신 해결해 주어, 개발자가 핵심 비즈니스 로직 개발에만 집중하도록 지원
        - 필요한 기능들이 부품처럼 미리 규격화되어 있어서, 약속된 규칙에 맞춰 조립하듯이 빠르게 대규모 웹 서비스 시스템 신속 구축
        - 데이터베이스 종류나 연동하는 외부 시스템의 기술 규격이 도중에 바뀌더라도, 이미 작성해 둔 프로그램 코드를 고치지 않고 부품만 유연하게 교체
    - 단점
        - 제공하는 기술 범위가 광범위하고 핵심 개념이 추상적이며, 초보 개발자가 구동 원리를 완벽히 파악하기 전까지 학습 장벽이 존재함
        - 프로젝트를 시작할 때 수많은 구성 파일이나 라이브러리 설정을 수동으로 정밀하게 제어해야 하므로, 초기 환결을 구축하는 절차가 매우 복잡함
    - 과거 자바 진영의 기술 규격이엇던 EJB(Enterprise JavaBeans)는 보안과 분산 처리 기능을 제공했으나, 간단한 기능 하나를 만들 때도 수많은 설정 파일 작성과 무거운 인터페이스 상속을 강제하여 개발 생산성이 저하되는 원인으로 지목

      [[Spring] Spring vs EJB](https://yummy0102.tistory.com/550)

    - 스프링은 기술 종속성 없이 가볍고 순수한 자바 기본 객체인 POJO(Plain Old Java Object)만을 사용하여 유연한 객체지향 설계 지원
- 스프링 핵심 용어 사전
    - 의존성 (Dependency)
        - 한 객체가 자신의 역할을 수행하는 과정에서 다른 객체를 호출하거나 참조하여 작동을 의지하는 협력 관계
        - 예를 들어 운전자(Driver) 객체가 정상 동작하기 위해 자동차(Car) 객체를 필요로 하는 것처럼, 하나의 컴포넌트가 작동하기 위해 필수적으로 결합되어야 하는 대상 객체나 리소스
    - **의존성 주입 (Dependency Injection)**
        - 어떤 객체가 필요로 하는 협력 객체를 스스로 직접 생성하여 확보하지 않고, 외부의 관리 장치(컨테이너)로부터 주입(전달)받아 사용하는 설계 기법
        - 객체 간 결합도를 낮추고 모듈 독립성을 높여 기술의 사후 변경을 용이하게 유도
    - 스프링 빈 (Spring Bean)
        - 스프링 컨테이너가 스스로 생성, 관계 조립, 생명주기 관리를 전담하는 대상 자바 객체
        - 개발자가 직접 new 연산자로 선언하여 사용하는 일반 자바 객체로 구별되며, 컨테이너에 의해 제어권이 위임된 컴포넌트만을 지칭
    - 스프링 컨테이너 (Spring Container / IoC Container)
        - 스프링 빈의 인스턴스화, 구성설정, 생명주기 제어 및 의존 관계 연결 처리를 담당하는 핵심 구동 엔진
        - 개발자가 작성한 컴포넌트들을 유기적으로 조립하는 자동화 공장 역할을 수행
    - 설정 메타데이터 (Configuration Metadata)
        - 스프링 컨테이너가 어떤 객체를 생성하고 어떻게 의존 관계를 이어 붙여 조립할지 지시하는 조립 설계도 문서
        - 과거에는 XML 문서 형태로 기재했으나, 현대 스프링 표준에서는 자바 설정 클래스 파일 상단에 Configuration, Bean 등의 어노테이션 지시어를 부여하여 관리
- 빌드와 배포
    - 빌드의 정의
        - 개발자가 작성한 소스코드를 컴퓨터가 실행할 수 있는 최종 소프트웨어 산출물로 변환하는 일련의 작업 과정
    - 빌드의 세부 과정
        - 의존성 해결 (Dependency Resolution): 빌드 설정 파일(build.gradle 등)을 파싱하여 프로젝트 가동에 필요한 외부 라이브러리들을 원격 저장소에서 다운로드
        - 컴파일 (Compile): 사람이 읽기 쉬운 자바 소스코드(.java)를 JVM이 실행할 수 있는 바이트코드(.class)로 번역
        - 테스트 실행 (Test): JUnit 등의 테스트 프레임워크를 가동하여 작성된 비즈니스 로직의 결함 유무를 빌드 완료 전에 최종 검증함
        - 패키징 (Packaging): 컴파일을 마친 바이트코드와 리소스 파일(propeties, yml 등)을 단일 실행 가능한 형태(.jar)로 압축함
    - 배포(Deploy)의 정의
        - 빌드를 통해 생성된 최종 소프트웨어 산출물을 실제 서비스를 제공할 운영 서버 환경에 올리고 실행하여 사용자가 서비스를 이용할 수 있도록 만드는 과정
- 스프링 프로젝트의 빌드와 배포 절차
    - 소스코드 작성: 개발자가 로컬 PC에서 요구사항에 맞는 자바 프로그램 소스코드와 애플리케이션 설정 파일을 작성함
    - 빌드 도구 가동: Gradle 또는 Maven 빌드 도구를 실행하여 작성된 소스코드를 컴파일하고 오류 검증용 테스트를 통과시킨 뒤, 최종 실행 파일로 압축 패키징함
    - 실행 파일 전송: 로컬 PC 빌드로 산출된 .jar 실행 파일을 실제 서비스를 운영할 원격 서버나 클라우드 서버 환경으로 전송함
    - 배포 및 service 실행: 전송받은 서버 환경에서 java -jar 명령어로 애플리케이션을 단독 구동하여 사용자의 실제 요청을 처리할 수 있도록 서비스를 개시함
- 빌드 도구: Maven과 Gradle
    - Maven
        - 아파치 재단에서 개발한 자바 프로젝트 빌드 도구, XML(pom.xml) 기반의 정형화된 라이프사이클 모델을 따름
        - 특징: 프로젝트 설정과 의존성 정보를 XML 형태로 엄격하게 표현하며, 설정 방식이 규격화되어 일관된 관리를 할 수 있음
        - 한계: 프로젝트 규모가 커질수록 XML 구조가 비대해지고 태그 중복이 증가하여 가독성이 저하되며, 빌드 스크립트 내 동적 논리 제어가 어려움
    - Gradle
        - Groovy 또는 Kotlin DSL(Domain Specific Language) 설정 언어를 탑재하여, 메이븐의 선언적 설정 장점과 프로그래밍 유연성을 결합한 표준 빌드 도구
        - 특징: 단순 설정을 넘어 빌드 스크립트 내에서 변수 선언 및 제어문 활용 등 동적인 빌드 스크립트 작성 지원
        - 성능적 장점: 이전 빌드 결과물을 재사용하고, 변경이 발생한 소스 파일만 선별적으로 컴파일하는 증분 빌드 기법을 적용하여 빌드 속도를 개선
    - 의존성 관리 (Dependency Management)
        - 설정 파일에 라이브러리 명칭만 명시하면 빌드 도구가 네트워크를 통해 필요한 라이브러리 자동 다운로드

## 4~6교시: 스프링 프로젝트 설정 방법과 상세, IoC, DI

- build.gradle 기본 설정 및 구조 분석
    - gradle 기반 자바 프로젝트를 생성하면 다음과 같은 표준 build.gradle 빌드 설정 파일이 자동 생성됨

    ```java
    plugins {
        id 'java'
    }
    
    group = 'net.likelion.bebc25'
    version = '1.0-SNAPSHOT'
    
    repositories {
        mavenCentral()
    }
    
    dependencies {
        testImplementation platform('org.junit:junit-bom:6.0.0')
        testImplementation 'org.junit.jupiter:junit-jupiter'
        testRuntimeOnly 'org.junit.platform:junit-platform-launcher'
    }
    
    test {
        useJUnitPlatform()
    }
    ```

    - plugins
        - 프로젝트 빌드 프로세스에 필요한 추가 기능과 도구를 gradle에 적용하는 영역
        - `id ‘java’` 는 gradle이 자바 프로젝트를 인식하여 컴파일 및 빌드할 수 있는 기본 기능을 제공
    - group, version
        - 빌드된 산출물(JAR 등)을 공유하거나 배포할 때 구분하기 위한 고유 식별 정보와 버전을 명시
    - repositories
        - 프로젝트에 필요한 외부 라이브러리들을 받아올 서버 저장소를 지정
        - `mavenCentral()` 은 메이븐 중앙 라이브러리 저장소를 지칭
    - dependencies
        - 프로젝트 컴파일 및 테스트 구동에 사용할 외부 라이브러리들을 gradle이 다운로드하도록 선언하는 영역
        - 기본적으로 단위 테스트들을 위한 junit-jupiter 라이브러리가 명시되어 있음
    - test
        - 테스트 실행 환경을 지정하는 태스크 설정 영역
        - `useJUnitPlatform()` 을 설정하여 JUnit 5 테스트 엔진으로 테스트가 실행되도록 지정
- 스프링 프레임워크 의존성 추가
    - 순수 자바 gradle 프로젝트에서 스프링의 핵심 원리인 IoC와 DI를 가동하기 위해서는 스프링 프레임워크 라이브러리를 build.gradle 설정에 수동으로 선언해 주어야 함
    - spring-context 의존성 선언
        - Spring Bean 컨테이너(ApplicationContext)를 구동하고, 객체의 자동 생명주기 및 의존 객체 조립 기능을 지원하는 spring-context 라이브러리를 build.gradle의 dependencies 블록 내에 선언

        ```java
        dependencies {
            implementation 'org.springframework:spring-context:7.0.8'
            testImplementation platform('org.junit:junit-bom:6.0.0')
            testImplementation 'org.junit.jupiter:junit-jupiter'
            testRuntimeOnly 'org.junit.platform:junit-platform-launcher'
        }
        ```

    - 의존성 로드 및 빌드 갱신
        - 방법 1: 편집창 단추의 새로고침 아이콘 클릭
        - 방법 2: 우측 가장자리의 Gradle 탭 창을 활성화 후 새로고침을 클릭
        - 다운로드 확인: gradle이 메이븐 중앙 저장소로부터 spring-context 라이브러리와 이에 연동되는 스프링 코어 모듈들을 자동으로 다운로드하여 컴파일 클래스 패스에 정상 적재하는지 최종 확인
            - Project 창의 SpringCore 프로젝트 아래에 External Libraries를 확장해서 `Gradle: org.springframework:spring-xxx:7.0.8` 라이브러리 들이 추가 되었는지 확인
    - 프로젝트 구조
        - src/main/java: 자바 소스 파일들이 위치하는 비즈니스 영역
        - src/main/resources: 설정 파일, 정적 리소스 등 자바 소스코드 이외의 파일 저장
        - src/test/java: 애플리케이션 동작을 검증하는 테스트 소스코드 작성 영역
        - build.gradle: 의존 라이브러리를 기재하고 플러그인을 명시하는 빌드 설정 파일
- 의존 객체 생성 구조와 제어의 역전 (IoC)
    - 의존성을 직접 결합하는 방식

    - 의존성을 느슨하게 결합하는 방식

        ```java
        interface Car {
          void startEngine();
          void drive();
          void stopEngine();
        }
        
        class GasolineCar implements Car {
          @Override
          public void startEngine() {
            System.out.println("가솔린 엔진에 점화하여 시동을 켭니다.");
          }
          @Override
          public void drive() {
            System.out.println("가솔린 동력으로 주행합니다.");
          }
          @Override
          public void stopEngine() {
            System.out.println("가솔린 공급을 차단하여 시동을 끕니다.");
          }
        }
        
        class HybridCar implements Car {
          @Override
          public void startEngine() {
            System.out.println("시스템 전원을 인가하고 하이브리드 시동을 켭니다.");
          }
          @Override
          public void drive() {
            System.out.println("전기와 엔진을 혼합하여 효율적으로 주행합니다.");
          }
          @Override
          public void stopEngine() {
            System.out.println("하이브리드 시스템 종료 처리를 작동합니다.");
          }
        }
        
        class Driver {
          private Car car; // 인터페이스에 의존 (느슨한 결합)
        
          // 생성자를 통해 외부에서 Car 구현체를 주입 받음
          public Driver(Car car) {
            this.car = car;
          }
        
          public void driveCar() {
            car.startEngine();
            car.drive();
            car.stopEngine();
          }
        }
        ```

        - 새로운 클래스 (예: ElectricCar)를 정의하여 구현하고, 외부에서 주입해 주는 객체만 바꿔 끼워주면 운전자 코드는 그대로 재사용함
    - 제어의 역전 (IoC, Inversion of Control)
        - 제어 주도권의 양도
            - 직접 결합 방식에서는 개발자가 Driver 클래스 내부에서 직접 new GasolineCar() 등을 기재하여 협력 객체의 생성과 의존성 조합을 스스로 통제함
            - 느슨한 결합 방식을 적용하면 Driver 클래스는 단지 Car 인터페이스에만 의존하며, 구체적인 구현 객체의 생성과 주입 권한은 외부로 넘어감
            - 이처럼 프로그램 흐름의 통제권 및 객체 생명주기 관리 권한이 개발자로부터 외부(프레임워크, 컨테이너)로 넘어가는 현상을 제어의 역전(IoC)이라고 함
        - 스프링 프레임워크에서의 IoC
            - 개발자가 XML 또는 자바 설정 클래스에 사용할 객체에 대한 정보를 기재해 두면, 스프링 컨테이너가 해당 정보를 바탕으로 컴포넌트(객체)들을 직접 생성하고 의존성을 주입하여 프로그램 흐름을 제어함
- 의존성 주입 (DI, Dependency Injection)
    - 객체가 스스로 자기에게 필요한 협력 객체(의존성)를 new 연산자로 선언해 만들지 않고, 외부에서 생성된 객체를 전달받아 사용하는 객체지향 설계 기법
    - 의존성을 제공하는 주체(스프링 컨테이너)와 의존성을 사용하는 객체 간의 결합도를 낮추는 효과를 제공
    - 설정 클래스(AppConfig) 구현: Bean 생성을 전담 제어할 수 있는 설정 클래스를 정의

        ```java
        @Configuration // 스프링 컨테이너 설정 정보 클래스 정의
        public class AppConfig {
          
          @Bean // 스프링 빈 등록 명시 (메서드명인 car가 빈의 고유 이름이 됨)
          public Car car() {
            return new GasolineCar(); // 필요 시 new HybridCar() 로 손쉽게 변경 가능
          }
        
          @Bean // driver 빈 등록 및 의존성 수동 전달
          public Driver driver(Car car) { // 1차로 Car 타입의 빈을 찾고, 매개변수명 car라는 이름의 빈을 찾음
            return new Driver(car); // 위에서 등록된 car 빈을 직접 주입받아 객체 생성
          }
        }
        ```

    - 스프링 컨테이너 구동 및 Bean 조회
        - 메인 실행 클래스에서 프레임워크가 제공하는 어노테이션 설정 클래스 기반 컨테이너(`AnnotationConfigApplicationContext`)를 사용하여 설정 정보를 로딩하고 조립된 Bean을 검색해 실행하는 흐름 구성

        ```java
        public class SpringCoreApplication {
          public static void main(String[] args) {
            // 1. 순수 스프링 프레임워크의 자바 설정 컨테이너를 수동 구동 (AppConfig 설정 정보 적재)
            AnnotationConfigApplicationContext context = new AnnotationConfigApplicationContext(AppConfig.class);
        
            // 2. 스프링 컨테이너에서 최종 완성 조립된 Driver 빈 조회
            Driver driver = context.getBean(Driver.class);
        
            // 3. 비즈니스 기능 가동
            driver.driveCar();
        
            // 4. 컨테이너 종료 및 리소스 반환
            context.close();
          }
        }
        ```
