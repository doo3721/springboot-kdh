## 2교시: 의존성 주입 방식

- 의존성 주입 방식
    - 생성자 주입 (Constructor Injection)
        - 객체 생성 시점에 생성자를 통해 필요한 의존 관계를 주입받는 방식
        - 장점
            - 객체 생성 시 딱 한 번만 호출되므로 런타임에 의존 객체가 바뀔 위험이 없어 불변성(immutability) 확보 가능
            - 자바 final 키워드 선언할 수 있어 의존성 주입 누락 시 컴파일러 단계에서 에러로 검증 가능
            - 순수 자바 코드로 단위 테스트 작성 시 생성자에 mock 객체를 주입하기 용이함
        - 단점
            - 의존 관계가 가변적이거나 선택적인 경우(런타임에 유연하게 변해야 하는 경우) 대응하기 어려움
    - 수정자 주입
        - setter 메소드를 정의하여 필요한 의존 관계를 주입받는 방식
        - 장점
            - 의존 관계가 필수적이지 않거나 런타임에 유연하게 변경이 필요한 가변 관계에 적합함
        - 단점
            - setter 메소드가 public으로 노출되므로 의존 객체가 임의로 변경될 위험이 존재함
            - 주입을 누락해도 객체가 정상적으로 생성되어, 이후 비즈니스 호출 시 NullPointException 발생 위험이 높음
    - 필드 주입
        - 멤버 변수 상단에 의존 주입 어노테이션을 부착하여 스프링 컨테이너가 직접 값을 할당하도록 유도하는 방식
        - 장점
            - 별도의 생성자나 setter 작성 없이 어노테이션 한 줄로 의존 관계를 선언하므로 코드가 간결함
        - 단점
            - 스프링 컨테이너가 구동되지 않는 순수 자바 단위 테스트 환경에서 의존 객체를 수동 조립할 방법이 없어서 테스트 작성이 어려움
            - 프레임워크 기술 규격에 과도하게 밀착 종속되므로 컴포넌트 독립성이 훼손되는 안트 패턴에 해당함
    - 생성자 주입을 권장하는 이유
        - 객체의 불변성(Immutability) 보장: 의존 관계가 조립된 이후 런타임에 외부 공격이나 실수로 인해 주입 객체가 변경되는 사고를 컴파일 수준에서 원천 차단함
        - 컴파일 타임 의존성 검증: final 필드를 누락한 채 생성자를 구현하면 컴파일러가 오류를 잡아주므로 NullPointException를 완전 예방함
        - 순수 자바 테스트 용이성: 프레임워크(스프링 컨테이너) 구동 없이도 테스트 코드에서 수동으로 new Driver(mockCar) 처럼 순수 자바 객체를 조립해서 테스트가 가능함
    - 순환 참조 문제와 방지 대책
        - 두 개 이상의 객체가 서로를 멤버 필드로 품어 상호 의존하는 상태를 순환 참조라고 함
        - 순환 참조 발생 시 현상
            - 필드 주입이나 수정자 주입을 사용하면, 스프링 컨테이너 가동 및 객체 생성과 의존성 주입 단계에서는 문제가 없지만 비즈니스 메소드가 호출되어 서로를 순환 참조할 때 비로소 스택오버플로우(StackOverflowError)를 발생시켜 런타임 에러를 초래함
        - 생성자 주입의 해결책
            - 생성자 주입을 채택하면 스프링 컨테이너 구동(인스턴스 생성 및 Bean 등록) 단계에서 생성자가 상대 객체를 서로 필요로 하는 상황인 순환 참조 관계에 놓인 객체들을 감자하고 즉시 구동을 거부하여 `BeanCurrentlyCreationException` 을 발생시키므로 개발 환경에서 미리 예방 및 제거 조치가 가능함

## 3~4교시: AOP, 프록시 기반 AOP 기술

- 관점 지향 프로그래밍 (AOP, Aspect Oriented Programming)
    - 핵심 관심사와 횡단 관심사
        - 애플리케이션의 핵심 가치와 비즈니스 목적으로 나타내는 비즈니스 관심 영역을 핵심 관심사(Core Concern)라고 함
        - 핵심 관심사 이외에 여러 모듈과 클래스 전반에 걸쳐 공통적으로 흩어져 반복 수행되는 부가 기능(로깅, 보안, 성능 측정, 트랜잭션, 에러 처리 등)을 횡단 관심사(Cross-cutting Concern)라고 함
    - OOP만 사용할 경우의 문제점
        - 코드 중복성 증가: 동일한 횡단 관심사 코드가 여러 클래스와 메소드에 반복 분산 선언되면서 전체 프로젝트의 중복 코드 비중을 증가시킴
        - 코드 가독성 저하: 핵심 비즈니스 로직 사이에 공동 부가 기능 코드가 혼재되면서 가독성을 가로막고, 이로 인해 잠재적 결함 및 오류 식별에 어려움을 초래함
        - 개발 생산성 감소: 비즈니스 도메인 분석에 주력해야 할 애플리케이션 개발자가 인프라에 제어 목적의 공통 부가 코드를 매번 직접 작성해야 하므로 핵심 개발 업무의 집중력이 떨어짐
        - 코드 재사용성 저하: 공통 부가 로직이 특정 객체 내에 밀접하게 결합하여 작동하므로, 해당 기능만 따로 분리하여 다른 서비스 컴포넌트에 이식하기 어려움
        - 유지보수 난이도 증가: 새로운 횡단 관심 사항의 적용이나 기존 부가 로직의 변경이 발생할 때 여러 클래스를 전부 수정해야 하므로 변경 대응력이 하락함
    - AOP 도입을 통한 한계 극복
        - 객체지향 프로그래밍은 모듈화 설계에 우수하지만, 횡단 관심사와 핵심 관심사 코드를 원천적으로 분리 정돈하기 어려운 구조적 한계가 존재함
        - 이 같은 문제를 극복하기 위해 핵심 관심사와 횡단 관심사를 명확하게 분리하여 각각 독립적인 모듈로 구축하고 관리하는 설계 패러다임인 관점 지향 프로그래밍(AOP)을 도입함
- 프록시(Proxy) 기반 AOP 기술
    - 프록시(Proxy) 개념
        - 의존 객체를 주입할 때 실제 비즈니스 객체(Target)를 전달 하는게 아니라, 대리인 객체(Proxy)를 생성하여 전달
        - 대리인 객체가 호출을 받아 횡단 관심을 처리하고 진짜 객체를 호출해 주는 대행 메커니즘을 기반으로 AOP를 달성함
    - 스프링이 사용하는 두 가지 프록시 생성 기술
        - JDK Dynamic Proxy
            - 자바 리플렉션 API의 java.lang.reflect.Proxy 클래스를 사용하여 런타임에 프록시를 동적으로 생성하는 기법
            - 반드시 Target 객체가 인터페이스를 구현하고 있어야 적용 가능함
            - 스프링 프레임워크의 인터페이스 유무에 따른 프록시 생성 방식
                - 인터페이스가 존재하는 경우: JDK Dynamic Proxy 기술을 기본으로 사용하여 프록시 생성
                - 인터페이스가 없는 경우: CGLIB Proxy 기술을 적용하여 타깃 클래스를 상속받는 프록시 생성
                - 설정을 통해 인터페이스가 있더라도 CGLIB 프록시 적용 강제 가능 (proxyTargetClass 옵션 활성화)
            - JDK Dynamic Proxy 제약사항
                - 인터페이스 구현 필수: 인터페이스가 없는 구체 클래스는 동적 프록시 생성 불가
                - 타입 캐스팅 제한: 생성된 프록시 객체는 인터페이스 타입으로만 변환할 수 있어 구현 클래스 타입으로 선언된 bean 주입 시 오류 유발
        - CGLIB(Code Generation Library, 시지엘리브) Proxy
            - 외부 바이트코드 조작 라이브러리를 사용하여 타겟 클래스를 상속받는 프록시 객체를 하부에서 동적으로 복제 생성하는 기법
            - 인터페이스 유무와 무관하게 동작하므로, 현대 스프링 프레임워크와 스프링 부트의 기본 프록시 생성 기술로 공식 채택되어 사용됨
            - 스프링 부트의 프록시 생성 정책
                - 인터페이스 존재 여부와 무관하게 기본적으로 CGLIB Proxy 방식으로 프록시 생성
                - 인터페이스 타입 대신 구현 클래스 타입으로 bean을 주입받아 사용할 때 발생하는 타입 캐스팅 예외 예방 목적
            - CGLIB Proxy 제약사항
                - final 클래스 상속 불가: final 클래스는 상속을 통한 하부 프록시 객체 생성 불가능
                - final 메소드 재정의 불가: final 메소드는 오버라이딩 진행할 수 없어 공동 관심사 부가 기능 주입 불가

## 5~6교시: 스프링 AOP, AspectJ

- 스프링 AOP와 AspectJ의 관계
    - AspectJ
        - 자바 진영의 사실상 표준(de facto standard) AOP 프레임워크
        - 컴파일 타임이나 클래스 로딩 타임에 실제 자바 클래스 바이트코드를 수정하여 가동하므로 생성자 호출이나 필드 변경 등 모든 영역에 AOP 개입이 가능함
    - 스프링 AOP
        - AspectJ의 편리한 어노테이션(@Aspect, @Before 등) 및 포인트컷 문법을 차용하여 구현한 스프링 내부의 경량 AOP 솔루션
        - 스프링이 관리하는 객체(bean)의 메소드 호출 통로에 대리인(proxy)을 세워 가로채는 방식으로 동작하므로 간편하지만 메소드 호출 시점에만 AOP를 적용할 수 있다는 한계가 존재함
- AOP 핵심 용어
    - 관점 (Aspect)
        - 분리해 낸 횡단 관심사를 하나로 묶어 독립적으로 설계한 클래스
        - 클래스 선언부에 @Aspect 지정하여 구현함
    - 결합 지점 (JoinPoint)
        - Aspect가 삽입되어 동작할 수 있는 프로그램 실행 과정상의 지점이나 타이밍을 뜻함
        - 프록시 방식을 사용하는 스프링 AOP는 오직 Spring Bean의 메소드 실행 시점만을 JoinPoint로 지원함
        - 개발자가 코드상에서 임의로 지정하는 것이 아니며, Spring Bean의 모든 메소드 호출 시점이 곧 JoinPoint가 됨
    - 필터 지점 (Pointcut)
        - 수많은 JoinPoint 중에서 실제로 Aspect를 적용할 대상 지점들을 지정하는 필터링 패턴 규칙
        - 코드상에서 @Pointcut이나 Advice 어노테이션 표현식으로 기입하여 선언함
        - Pointcut 필터 패턴 종류
            - execution
                - 가장 대표적이고 널리 사용되는 메소드 실행 JoinPoint 매칭 지시자
                - 반환 타입, 패키지 경로, 클래스명, 메소드 시그니처, 파라미터 규격을 종합 매칭함
                - 예: `execution(* net.likelion..*Service.*(..))`
            - @annotation
                - 특정 어노테이션이 부착된 메소드 지점만을 한정 타겟팅하는 지시자
                - 핵심 비즈니스 로직에 침투하지 않는 비침투적 AOP 설계 시 유용하게 결합됨
                - 예: `@annotation(net.likelion.TimeTrace)`
            - within
                - 특정 타입이나 특정 패키지 하위의 모든 JoinPoint를 클래스/패키지 수준에서 넓게 매칭하는 지시자
                - 예: `within(net.likelion.bebc25.springcore.oop..*)`
            - bean: Spring Bean의 ID 또는 Bean 이름의 와일드카드 패턴을 매칭하는 스프링 전용 지시자
                - 예: `bean(*Car)`
            - this 및 target
                - 런타임에 형성된 프록시 객체의 타입(this) 또는 실제 Target 객체의 타입(target)을 기준으로 매칭 대상을 필터링하는 지시자
    - 대상 객체 (Target)
        - Aspect를 부여받는 실질적인 비즈니스 핵심 클래스 컴포넌트
        - 개발자가 코드상에서 어노테이션을 붙이거나 명시적으로 지정하지 않으며, Pointcut에 의해 필터링된 Spring Bean 객체들이 자동으로 Target이 됨
    - 부가 동작 (Advice)
        - Pointcut으로 지정된 대상 지점에서 실제로 수행할 구체적인 횡단 관심사(공통 기능) 구현 코드 블록
        - 코드상에서 @Before, @Around 등을 적용해 작성하며, 타깃 메소드의 실행 주기에 맞춰 다음과 같이 5가지 동작 시점을 지원함
        - Advice의 동작 지점
            - @Before (이전): 타깃 메소드가 실행되기 직전에 횡단 관심사를 가동하는 형태
            - @AfterReturning (정상 반환 이후): 타깃 메소드가 예외 없이 성공적으로 실행을 끝마치고 결과 값을 반환한 직후에 가동하는 형태
            - @AfterThrowing (예외 발생 이후): 타깃 메서드 실행 도중 예외가 발생하여 예외가 던져지는 시점에 가동하는 형태
            - @After (이후): 타깃 메소드의 성공 여부와 무관하게 메소드가 끝나면 무조건 실행되는 사후 처리 형태 (try-catch의 finally 블록과 유사함)
            - @Around (전후 주변): 타깃 메소드의 호출 전후 과정을 완전히 장악하여 실행 시점 통제, 반환값 조작, 예외 가로채기 등 가장 강력한 제어를 지원하는 형태

    ```java
    @Aspect // -> Aspect (무엇을: 공통 관심사 클래스 정의)
    @Component
    public class LoggingAspect {
    
        // -> Pointcut (어디에: 적용 대상 지점을 정의하는 필터 패턴)
        // -> Target (누구에게: 위 execution 지시 범위에 속하는 oop 패키지의 Driver, ElectricCar 등의 핵심 비즈니스 빈 객체들)
        // 아래 oopPackageMethods()는 실제 로직이 아니라, 포인트컷 표현식에 이름(식별자)을 부여하여 하단 @After 등에서 재사용할 수 있게 해주는 빈 메서드임 (바디가 비어 있음)
        @Pointcut("execution(* net.likelion.bebc25.spring.aop.*Driver.*(..))")
        private void oopPackageMethods() {}
    
        // -> JoinPoint (언제: 메서드 호출 이전인 @Before 타이밍에 개입)
        // -> Advice (언제 어떤 기능을: 메서드 바디 내부의 공통 로깅 출력 코드)
        // 아래 @Before 어노테이션 인자에 포인트컷 표현식을 인라인 문자열 상수로 직접 기입하여 사용함
        @Before("execution(* net.likelion.bebc25.spring.aop.*Driver.*(..))")
        public void logBefore() {
            System.out.println("[AOP 로그] 메서드 실행 전에 가동되는 공통 부가 기능");
        }
    
        // -> JoinPoint (언제: 메서드 호출 완료 이후인 @After 타이밍에 개입)
        // -> Advice (언제 어떤 기능을: 메서드 바디 내부의 공통 로깅 출력 코드)
        // 아래 @After 어노테이션 인자에 상기 선언해 둔 포인트컷 식별자 메서드명(oopPackageMethods)을 호출하여 재사용함
        @After("oopPackageMethods()")
        public void logAfter() {
            System.out.println("[AOP 로그] 메서드 실행 후에 가동되는 공통 부가 기능");
        }
    }
    ```
