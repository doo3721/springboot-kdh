## 1~3교시: 컨트롤러 복습 및 실습

### Spring MVC 컨트롤러 설계 및 웹 요청 핸들러 정의

- **Controller 어노테이션 정의 및 데이터 직접 응답**
    - HTTP 요청을 수신하여 비즈니스 로직을 연결하고, 그 결과를 클라이언트에게 직접 문자열이나 HTML 데이터로 반환하는 기초 웹 컨트롤러 설계 기법
    - **컨트롤러 정의 및 응답용 어노테이션**
        - @Controller
            - 클래스 상단에 명시하여 해당 클래스를 스프링 컨테이너가 관리하는 웹 요청 핸들러(컨트롤러) bean으로 등록함
        - @ResponseBody
            - 메소드 상단 혹은 반환 타입 앞에 부여하며, 메소드가 리턴하는 값(HTML 텍스트, DTO 등)을 HTTP 응답 메세지의 바디(Body) 영역에 직접 기입하여 전송함
        - ResponseEntity<T> 클래스
            - HTTP 응답의 3대 요소인 상태 코드 (Status), 응답 헤더(Headers), 응답 바디(Body)ㄹ르 자바 객체 형태로 정밀하게 제어하여 반환할 수 있게 스프링이 제공하는 전용 응답 객체
            - 일반 컨트롤러 메소드에서 DTO나 문자열을 단순히 리턴하면 응답 상태 코드가 무조선 `200 OK` 로 나가지만, ResponseEntity를 반환 타입으로 사용하면 상황에 맞게 `201 Created` 나 `400 Bad Request` 등 커스텀 상태 코드를 직접 지정해서 보낼 수 있음

            ```java
            // 사용법 예시: 응답 바디에 메시지를 실어 보내며, HTTP 상태 코드를 201 Created로 지정함
            @GetMapping("/api/ok")
            public ResponseEntity<String> successResponse() {
                return ResponseEntity
                        .status(HttpStatus.CREATED)
                        .body("데이터 저장 완료");
            }
            ```

    - **웹 요청 경로 매핑용 어노테이션 및 주요 속성**
        - @RequestMapping
            - 클래스나 메소드에 부여하여 해당 컨트롤러가 처리할 공통 URL 경로 범위를 지정
            - 특히 클래스 레벨에 선언할 경우, 내부 개별 메소드에 정의된 매핑 경로들의 앞부분에 공통 접두사(Prefix) 형태로 결합되어 URL 중복 선언을 방지하고 통합 제어하기 수월
            - 주요 속성
                - value 또는 path: 매핑할 URL 주소 경로를 지정
                - method: 처리할 HTTP 메소드 방식을 직접  지정 (생략할 경우 모든 메소드를 다 수용)

            ```java
            @RequestMapping(value = "/board/list", method = RequestMethod.GET)
            @ResponseBody
            public String getBoardList() {
                return "<html>...</html>";
            }
            
            // 클래스 레벨에 공통 접두사 경로(/board) 선언
            @Controller
            @RequestMapping("/board")
            public class BoardController {
            
                // 최종 요청 주소: /board/list (GET 방식)
                @RequestMapping(value = "/list.html", method = RequestMethod.GET)
                @ResponseBody
                public String getBoardList() {
                    return "<html>...</html>";
                }
            }
            ```

        - @GetMapping 및 @PostMapping
            - 각각 HTTP GET, POST 요청을 수신할 세부 URL 매핑 경로를 단축형으로 지정할 때
            - 주요 속성
                - value 또는 path: 세부 URL 주소 경로를 지정

            ```java
            @GetMapping(value = "/board/list")
            public String getBoardList() {
                return "<html>...</html>";
            }
            
            @PostMapping(value = "/board/write")
            public String writePost() {
                return "redirect:/board/list";
            }
            ```

            - 리다이렉트 (Redirect 방식)
                - 응답 헤더에 302 상태 코드와 Location 경로를 담아 전송하여 브라우저가 새 주소로 다시 GET 요청을 보내도록 유도함
                - 브라우저의 URL 주소창이 최종 목적지 경로로 변경되므로 새로고침 시 중복 처리(게시글 이중 등록 등) 문제를 제어하는 데 사용

## 4교시: HTTP 파라미터 매핑 방법, View 구현

- **HTTP 파라미터 매핑 방법**
    - 클라이언트가 전달한 요청 데이터를 컨트롤러 메소드의 매개변수로 전달 받는 방법
        - @RequestParam: HTTP 요청 파라미터(예: https://…/…?name=value)를 1대1 매핑하여 수집

            ```java
            // 요청 URL: /search?keyword=spring
            @GetMapping("/search")
            @ResponseBody
            public String search(@RequestParam("keyword") String keyword) {
                return "검색어: " + keyword; // 검색어: spring
            }
            ```

        - @PathVariable: URI 경로 자체에 포함된 변수 값(예: /posts/{id}에서 id)을 추출하여 바인딩

            ```java
            // 요청 URL: /posts/15
            @GetMapping("/posts/{id}")
            @ResponseBody
            public String getDetail(@PathVariable("id") int id) {
                return "조회된 번호: " + id; // 조회된 번호: 15
            }
            ```

        - @ModelAttribute: HTML 폼 데이터를 DTO 객체 필드에 일괄 바인딩(DTO의 기본 생성자 호출 후 setter를 이용해 값을 주입, 기본 생성자가 존재해야 함)

            ```java
            // HTML 폼에서 전송된 title, content, author 값을 PostDto 객체 필드에 자동 매핑함
            @PostMapping("/write")
            @ResponseBody
            public String writePost(@ModelAttribute PostDto post) {
                return "제목: " + post.getTitle();
            }
            ```


### 뷰(View) 구현 및 렌더링

- **서버 사이드 동적 화면 생성 기술의 변천사**
    - 서블릿(Servlet) 기반의 화면 렌더링
        - 자바 클래스 파일 내에 PrintWriter 객체를 이용해 HTML 마크업 문자열을 직접 하드코딩하여 응답 스트림으로 전송하던 방식
        - 단순 오타 수정이나 레이아웃 변경 시에도 매번 자바 코드를 컴파일하고 재배포해야 하므로 개발 생산성 저하 극심

        ```java
        @WebServlet("/users")
        public class UserServlet extends HttpServlet {
            @Override
            protected void doGet(HttpServletRequest req, HttpServletResponse resp) throws IOException {
                resp.setContentType("text/html;charset=UTF-8");
                PrintWriter out = resp.getWriter();
                
                List<User> users = getUsers();
                
                out.println("<html>");
                out.println("<head><title>사용자 목록</title></head>");
                out.println("<body>");
                out.println("  <h1>사용자 목록</h1>");
                out.println("  <ul>");
                for (User user : users) {
                    out.println("    <li>" + user.getName() + " (" + user.getEmail() + ")</li>");
                }
                out.println("  </ul>");
                out.println("</body>");
                out.println("</html>");
            }
        }
        ```

    - JSP(Java Server Pages) 기반의 화면 렌더링
        - HTML 문서 구조 안에 JSP 지시문(Directive, <%@ %>). 스크립틀릿(Scriptlet, <% %>, 표현식(<%= %>) 등 전용 태그 문법을 활용해 페이지 설정 정보를 정의하고 자바 프로그래밍 코드를 작성하고 변수값을 출력하는 작업을 통해 서블릿 대비 렌더링 효율 향상
        - JSP 파일은 런타임 시점에 WAS에 의해 자바 소스 파일로 자동 파싱된 후, 컴파일 과정을 거쳐 실제 서블릿 인스턴스로 변환 및 실행됨 (JSP는 서블릿 코드를 좀 더 HTML 친화적인 구조로 작성할 수 있도록 지원하는 대안)
        - 화면을 표현하는 프레젠테이션 영역과 비즈니스 로직이 단일 파일에 어지럽게 섞여 스파게티 코드가 되기 쉬우며, 디자이너와의 협업 난이도 대폭 상승(모델 1 방식)

        ```java
        <%@ page contentType="text/html;charset=UTF-8" language="java" %>
        <%@ page import="java.util.List, net.likelion.User" %>
        <%
            List<User> users = getUsers();
        %>
        <html>
        <head>
            <title>사용자 목록</title>
        </head>
        <body>
            <h1>사용자 목록</h1>
            <ul>
                <% for (User user : users) { %>
                    <li><%= user.getName() %> (<%= user.getEmail() %>)</li>
                <% } %>
            </ul>
        </body>
        </html>
        ```

    - MVC 패턴과 템플릿 엔진의 등장
        - 비즈니스 로직은 컨트롤러와 모델에 위임하고, 화면 렌더링은 데이터 주입 기능만 갖춘 템플릿 엔진에 위임하도록 명확히 분리
        - JSP가 가진 단점(반디스 WAS가 구동 중인 상태에서만 화면 확인 가능)을 극복하고, 로컬 환경에서 단독 HTML 렌더링이 가능한 모던 템플릿 엔진 기술로 발전
        - Thymeleaf 템플릿 엔진 코드 예시

            ```html
            <!DOCTYPE html>
            <html xmlns:th="http://www.thymeleaf.org">
            <head>
                <meta charset="UTF-8">
                <title>사용자 목록</title>
            </head>
            <body>
                <h1>사용자 목록</h1>
                <ul>
                    <!-- th:each와 th:text를 통해 동적 데이터를 매핑 및 출력 -->
                    <!-- 태그 바디의 텍스트(홍길동)는 서버 없이 브라우저로 단독 실행 시 출력되는 샘플 데이터이며, 서버 구동 시 실제 데이터로 대체됨 -->
                    <li th:each="user : ${users}" th:text="|${user.name} (${user.email})|">
                        하루 (haru@gmail.com)
                    </li>
                </ul>
            </body>
            </html>
            ```

- **스프링의 뷰(View)와 뷰 리졸버(ViewResolver) 아키텍처**
    - 스프링 웹 MVC 화면 렌더링 책임을 컨트롤러로부터 완전히 분리하기 위해 뷰(View)와 뷰 리졸버(ViewResolver)라는 인터페이스 기반의 추상화 설계를 도입함
    - 컨트롤러는 어떠한 화면 기술(JS, Thymeleaf, Mustache 등)이 배후에서 돌아가는지 알 필요 없이 단순히 화면의 논리적 명칭만을 반환하도록 하여 결합도를 느슨하게 격리함
- **뷰 리졸버(ViewResolver) 인터페이스**
    - 컨트롤러가 반환한 뷰의 논리적 식별 이름(예: “board/list”)을 해석하여 실제 브라우저 응답 처리가 가능한 물리 뷰 객체(View)를 결정하고 찾아내는 책임을 담당하는 인터페이스
    - 주요 특징 및 탐색 순위 체인
        - 스프링 MVC는 여러 종류의 뷰 리졸버들을 빈으로 동시 등록할 수 있으며, 이들은 내부적인 우선순위(Order) 순서에 맞춰 등록 순서대로 체인을 이루어 매핑 파일을 스캔함
        - 주요 구현체
            - InternalResourceViewResolver: JSP 템플릿 파일 경로를 해석하여 서빙할 때 사용
            - ThymeleafViewResolver: 타임리프 템플릿 파일을 해석하여 서빙할 때 사용
            - FreeMakerViewResolver, MustacheViewResolver: 각각의 전용 엔진 파일 경로를 해석하여 서빙할 때 사용
- **뷰(View) 인터페이스**
    - 컨트롤러로부터 전달된 데이터(Model)와 클라이언트에게 전달할 최종 응답 자원(HTML 문서, JSON 문자열 등)을 실제로 그려내는(렌더링) 역할을 수행하는 인터페이스
        - 예: JSP 화면을 그리는 InternalResourceView, 타임리프 화면을 그리는 ThymeleafView 등
    - 핵심 메소드
        - `void render(Map<String, ?> model, HttpServletRequest request, HttpServletResponse response) throws Exception;`
        - 이 렌더링 메소드가 호출되면 각 View 구현체들은 모델 내부에 담긴 데이터를 읽어들여 템플릿 문서 상의 동적 위치에 바인딩 처리를 완료한 후 응답 스트림에 출력 데이터를 전달함
- **뷰 처리 렌더링 동작 시나리오**
    - 컨트롤러가 비즈니스 연산을 완수한 후, 화면에 주입할 데이터(Model)와 이를 처리할 뷰 이름(예: “board/list”)을 디스패처 서블릿(DispatcherServlet)에 반환함
    - 디스패처 서블릿은 등록된 뷰 리졸버(ViewResolver) 체인을 높은 순서(Order)부터 차례대로 호출하여, 전달받은 뷰 이름에 대응하는 뷰(View) 객체의 생성을 의뢰함
        - ThymeleafViewResolver의 경우 “templetes/” + “board/list” + “.html” 경로를 조합하여 실제 템플릿 파일을 찾아내어 뷰 객체를 생성함
    - 뷰 리졸버가 특정 경로상의 물리 리소스를 매칭하여 뷰 객체(예: ThymeleafView 등)를 확보하여 디스패처 서블릿에 반환함
    - 디스패처 서블릿은 획득한 뷰 객체의 render() 메소드를 호출하며 매개변수로 모델 맵(Model Map)과 서블릿 응답 스트림 객체를 넘겨줌
    - 뷰 객체가 모델 데이터를 주입받아 완성한 최종 웹 리소스를 생성하고, 브라우저가 응답 스트림 객체를 이용해 출력을 수행함

## 5~6교시: Thymeleaf

### Thymeleaf 뷰 템플릿 엔진

- **타임리프 기본 문법**
    - https://www.thymeleaf.org/ 참고
    - HTML 코드의 본래 마크업 규격을 유지하면서 서버의 스프링 데이터를 주입하여 화면을 렌더링하는 내추럴 템플릿(Natural Templeates)엔진
    - 컨트롤러 데이터 바인딩 및 템플릿 파일 경로 규칙
        - 컨트롤러와 Model 규칙
            - 컨트롤러는 클라이언트 요청을 수신하면 스프링으로부터 Model 객체를 전달 받음
            - Model 객체는 뷰 템플릿과 바인딩시켜서 최종 뷰를 만들기 위한 데이터 역할을 함
        - `model.addAttribute(key, value)`
            - 뷰가 동적으로 렌더링할 때 꺼내어 쓸 수 있도록 지정한 key 명칭으로 실제 자바 객체(value)를 모델에 보관
            - 이후 뷰 템플릿에서 key 값으로 사용 가능
        - 물리적 템플릿 파일 저장 위치
            - 스프링 부트 웹 프로젝트에서 타임리프 템플릿 파일의 표준 기본 저장 경로는 `src/main/resources/templates/` 디렉토리
        - 파일 매핑 및 흐름
            - 컨트롤러에서 논리적 뷰 명칭으로 `“board/detail”` 을 문자열로 최종 반환
            - 스프링 MVC의 뷰 리졸버가 결합되어 실제 물리 파일 경로인 `src/main/resources/templates/board/detail.html` 파일을 찾아내 동적 데이터 바인딩 렌더링을 시작함
    - 타임리프 속성을 사용하려면 HTML 파일 최상단 html 태그에 네임스페이스(`xmlns:th="http://www.thymeleaf.org"`)를 명시해야 함

  #### 변수값 출력

    - 서버의 컨트롤러에서 Model 객체에 담아 전달한 변수값을 타임리프 뷰에서 꺼내어 태그 내부의 텍스트로 안전하게 주입함
    - 변수 표현식 `${...}` 을 사용해 Model에 저장된 key 이름으로 객체를 참조함

    ```java
    @GetMapping("/detail.html")
    public String getDetail(Model model) {
        PostDto post = new PostDto(1, "자바 공부", "스프링 웹 MVC 학습", "하루", LocalDateTime.now());
        model.addAttribute("post", post); // Model 객체에 post라는 key로 DTO 데이터 추가
        return "post/detail"; // 뷰 이름 반환
    }
    ```

    ```html
    <!-- Model에 담긴 post 객체의 title 필드값(자바 공부)을 출력함 -->
    <span th:text="${post.title}">샘플 게시글 제목</span>
    ```

  #### 조건문 사용하기

    - 지정한 조건식의 참 또는 거짓 여부에 따라 특정 HTML 요소를 화면에 선택적으로 노출
    - th:if(참일 때 노출) 및 th:unless(거짓일 때 노출) 속성을 조합하여 제어

    ```html
    <!-- 작성자 정보가 존재할 경우 작성자 이름을 출력함 -->
    <span th:if="${post.author != null}" th:text="'작성자: ' + ${post.author}">작성자: 하루</span>
    
    <!-- 작성자 정보가 존재하지 않을 경우 익명으로 대체 출력함 -->
    <span th:unless="${post.author != null}">작성자: 익명</span>
    ```

  #### 반복문 사용하기

    - 리스트나 배열 등 컬렉션 데이터를 순회하며 동일한 HTML 마크업 요소를 반복하여 렌더링함
    - 반복 변수와 변수 표현식( `${ }` )을 조합하여 루프를 구현함

    ```html
    <tr th:each="post : ${posts}">
      <td th:text="${post.id}">1</td>
      <td th:text="${post.title}">게시글 제목</td>
    </tr>
    ```

  #### 링크 사용하기

    - 서버 배포 설정에 따른 컨텍스트 경로(예: server.servlet.context-path로 지정된 서브 경로)를 런타임에 자동으로 감지하고 결합하여 정적/동적 URL 링크를 안전하게 조립함
    - 하드코딩된 상대 또는 절대 경로 사용 시 배포 경로 변경에 따라 링크가 깨지는 문제를 방지하며, 배포 독립성과 이식성을 제공함
    - 링크 표현식 `@{...}` 과 함께 사용하며, 괄호 `(...)` 를 이용해 결로 변수를 주입하거나 쿼리 파라미터를 추가 지정할 수 있음

    ```html
    <a th:href="@{/board/posts/{id}/edit(id=${post.id})}">수정하기</a>
    ```

  #### 유틸리티 객체 사용하기

    - 날짜 형식 변환, 문자열 처리, 숫자 포맷팅 등 템플릿 내부에서 자주 수행하는 헬퍼 연산을 처리해 주는 타임리프 내장 유틸리티 객체를 제공

    ```html
    <!-- 날짜 및 시간 포맷팅 (#temporals) -->
    <span th:text="${#temporals.format(post.createdAt, 'yyyy-MM-dd HH:mm:ss')}">2026-07-13 18:00:00</span>
    
    <!-- 숫자의 천 단위 콤마 포맷팅 (#numbers) -->
    <span th:text="${#numbers.formatInteger(post.views, 3, 'COMMA')}">1,250</span>
    ```

  #### 폼 데이터 바인딩

    - 서버에서 전달받은 객체(DTO)의 필드와 웹 브라우저 화면의 입력 창(input)을 1대1로 연결하여 값을 서로 주고받을 수 있게 연동해 주는 기능
    - 입력 창에 데이터를 채워 서버로 전송하면 지정된 DTO 객체의 필드에 값이 자동으로 담기며, 반대로 DTO 객체에 담겨 있던 기본값을 화면 입력 창에 자동으로 띄워 보여줄 수도 있음
    - 주요 속성 및 바인딩 문법
        - `th:object`: 폼데이터의 기준이 되는 DTO 객체(Model에 담긴 객체)를 바인딩함
        - `th:field`: DTO 객체의 개별 필드를 input 입력 창과 1대1로 연결하며, 브라우저가 화면을 띄울 때 필요한 HTML 기본 속성(id, name, value)들을 백그라운드에서 자동으로 생성하고 채워 줌
        - 선택 변수 식 `*{필드명}`: `th:object` 로 지정한 객체의 내부 필드를 참조할 때 변수 표현식(`${모델명.필드명}`) 대신 간단히 단축 참조할 수 있는 문법
        - 예

        ```html
        <form th:action="@{/board/write}" th:object="${postForm}" method="post">
          <input type="text" th:field="*{title}" placeholder="제목을 입력하세요">
        </form>
        ```

        - 실제 브라우저 렌더링 결과물 (`th:action, th:object, th:field`가 표준 속성으로 변환 처리됨)

        ```html
        <form action="/board/write" method="post">
          <input type="text" id="title" name="title" value="" placeholder="제목을 입력하세요">
        </form>
        ```

    #### 리터럴 대체 문법
    
    - 변수 표현식과 일반 정적 문자열을 문자 결합 연산자( + )없이 편리하게 결합하여 사용할 수 있도록 제공하는 편의 문법
    - 전체 결합 문자열을 파이프( | ) 기호로 감싸서 표현하며, 가독성 향상 및 싱글 쿼테이션( ‘ ) 사용 번거로움 해소에 기여
    
    ```html
    <!-- 파이프 기호를 사용하지 않은 방식 (더하기 연산자와 싱글 쿼테이션 수동 결합) -->
    <span th:text="${user.name} + ' (' + ${user.email} + ')'">샘플 텍스트</span>
    
    <!-- 파이프 기호를 사용한 방식 (간결하고 직관적인 결합) -->
    <span th:text="|${user.name} (${user.email})|">샘플 텍스트</span>
    ```
    
    #### 타임리프 주요 표현식 기호 비교
    
    - 템플릿 내부에서 데이터를 가리키는 기호에 따라 꺼내는 자원(객체, 다국어 메세지 파일, 내장 라이브러리 등)이 완전히 달라지므로 목적에 맞게 구별하여 지정
    - 주요 기호 및 용도
        - `${...}`(변수 표현식): 컨트롤러에서 Model에 담아 전달한 일반 자바 객체나 변수의 값을 직접 꺼낼 때 사용
        - `*{...}`(선택 변수 식): `th:object`로 지정해 둔 부모 객체의 내부 필드명을 생략형으로 단축 참조할 때 사용
        - `#{...}`(메세지 표현식): 다국어 지원을 위해 외부 메세지 설정 파일(properties)에 정의된 텍스트를 꺼내올 때 사용
        - `#객체명`(유틸리티 객체 접두사): 중괄호 없이 쓰이며, 날짜 형식 변환이나 검증 필드 에러 검출 등을 처리하는 타임리프 기본 내장 객체를 가리킴
            - 예: #temporals, #numbers, #fields
