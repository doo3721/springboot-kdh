## 1교시: 복습

## 2교시: 컴포넌트 스캔, 의존관계 자동 주입

- 컴포넌트 스캔 (Component Scan)
    - 수동으로 @Configuration 클래스 내부에 @Bean 어노테이션을 부여하여 Spring Bean을 일일히 등록하는 방식은 등록할 Bean의 개수가 많아질수록 설정정보가 비대해지고 누락되는 실수가 많아짐
    - 스프링은 이러한 설정 파일의 번거로움을 생략하고, 특정 패키지 하위의 클래스들을 검색하여 @Component 어노테이션이 붙은 클래스를 spring bean으로 자동 등록해 주는 컴포넌트 스캔 기능을 제공
    - 스프링 설정 클래스(AppConfig)에 `@ComponentScan(basePackages = "패키지")` 어노테이션을 선언하면 지정한 패키지 범위 내의 클래스들을 검색하여 @Component 어노테이션이 부착된 컴포넌트들을 자동으로 bean으로 등록함.
    - @ComponentScan 어노테이션에 패키지 지정을 생략하면 설정 파일이 있는 패키지와 하위 모든 패키지가 컴포넌트 스캔의 대상이 됨
    - bean의 이름은 클래스명의 첫 글자를 소문자로 변환하여 지정됨
        - 예: HybridCar 클래스의 bean - hybridCar

    ```java
    @Component
    public class GasolineCar implements Car { }
    
    @Component
    public class HybridCar implements Car { }
    
    @Component
    public class Driver { }
    
    @Configuration
    @EnableAspectJAutoProxy
    @ComponentScan(basePackages = "net.likelion.bebc25.spring.aop.springaop")
    public class AppConfig {
        // 컴포넌트 스캔에 의해 @Component가 부착된 GasolineCar, HybridCar, Driver 등이 자동으로 등록되고 결합되므로 @Bean을 등록할 필요가 없음
    }
    ```

- 의존관계 자동 주입 (Dependency Auto Injection)
    - 각 클래스 내부에서 필요한 의존성 주입을 위해 @Autowired 어노테이션을 사용
    - 주입할 매개변수의 타입(인터페이스 또는 클래스)을 기준으로 컨테이너 내의 동일 타입의 bean을 탐색함.
    - 타입 검색 결과 동일한 타입의 bean 후보가 여러 개 발견될 경우, 주입 대상을 결정할 수 없어 컨테이너 구동 단계에서 NoUniqueBeanDefinitionException 예외가 발생
    - 생성자 주입 (Constructor Injection)
        - spring bean 클래스의 생성자 상단에 @Autowired를 선언하면 스프링 컨테이너가 생성자 파라미터에 요구되는 bean 객체를 자동으로 매칭하여 주입
        - 생성자가 단 1개만 존재할 경우에는 @Autowired 어노테이션을 생략해도 자동으로 의존관계 주입이 처리됨.

        ```java
        @Autowired
        public Driver(Car car){
            this.car = car;
        }
        ```

    - 수정자 주입 (Setter Injection)
        - 필드 값을 변경하는 수정자(setter) 메서드에 @Autowired를 선언하여 의존 관계를 주입받는 방식

        ```java
        @Autowired
        public void setCar(Car car){
            this.car = car;
        }
        ```

    - 필드 주입 (Field Injection)
        - 자바 클래스의 필드 변수 상단에 @Autowired 어노테이션을 붙여서 스프링 컨테이너가 의존성을 직접 주입하도록 하는 방식

        ```java
        @Autowired
        private Car car;
        ```

    - 동일 타입의 다중 Bean 자동 주입 제어
        - 동일한 인터페이스(Car)를 구현한 여러 개의 하위 클래스가 스프링 컨테이너에 동시에 bean으로 등록되어 있을 경우, 자동 의존 주입 단계에서 어떤 구현 객체를 매핑하여 사용할지 결정하는 제어 기법
        - 동일한 타입의 여러 빈이 있을 경우 예외 발생
            - 타입(Car)만을 기준으로 자동 주입(@Autowired)을 가동하면, 스프링은 대상 후보가 복수 개(GasolineCar, ElectricCar)여서 어떤 빈을 매칭할지 결정하지 못하고 예외(NoUniqueBeanDefinitionException)를 발생시킴.
        - 이 충돌 문제를 해결하기 위해 다음 제어 방식을 적용할 수 있음.
        - @Primary 어노테이션 기본값 선언
            - 동일한 타입의 bean 중 우선 주입 대상을 기본값으로 선언하는 방법
            - 특정 bean 클래스 상단에 @Primary 어노테이션을 설정하면, 호출하는 코드에서 별도로 bean 이름을 기입하지 않더라도 @Primary가 부착된 기본 bean을 최우선적으로 찾아 주입함

            ```java
            @Component
            @Primary // GasolineCar가 Car 타입의 Primary가 됨
            public class GasolineCar implements Car { }
            ```

            ```java
            @Component
            public class HybridCar implements Car { }
            ```

        - @Qualifier 어노테이션 지정
            - 생성자나 setter 메서드의 매개변수 앞에 @Qualifier 어노테이션을 부착하고 대상 bean의 이름(id)을 문자열로 선언하여 주입 관계를 명시적으로 지정

            ```java
            @Autowired
            public Driver(@Qualifier("hybridCar") Car car){
                System.out.println("생성자를 통해 외부에서 Car 구현체를 주입 받음");
                this.car = car;
            }
            ```

        - @Primary와 @Qualifier의 우선순위
            - 동일한 타입의 다중 bean 후보 중에서 @Primary와 @Qualifier가 충돌할 때의 처리 기준
            - @Primary는 기본 주입 대상을 제공하는 넓은 범위의 설정 방식이고, @Qualifier는 주입 지점에 대상을 구체적으로 지목하는 좁은 범위의 설정 방식
            - 스프링은 범위가 좁고 구체적인 설정에 더 높은 우선권을 부여하는 원칙을 따르므로, 상세 지정이 수행된 @Qualifier가 최우선으로 적용되어 주입됨

## 3~4교시: 스프링 컨테이너 핵심 메커니즘

### 스프링 컨테이너 핵심 메커니즘

- Spring Bean 스코프 (Scope)
    - Spring Bean 객체가 스프링 컨테이너 내부에서 유지되고 유효성을 가지는 생존 범위를 발함
    - 싱글톤 스코프 (Singleton Scope)
        - 컨테이너 구동 단계에서 bean을 단 하나만 사전 생성하여 전체 사용처에서 하나의 객체를 공유하는 스프링의 기본 스코프 방식
        - 상태를 저장하지 않는 무상태 방식으로 bean을 설계해야 멀티 스레드 가동 환경에서 안전한 동작을 보증할 수 있음
        - 상태를 공유하지 않고 비즈니스 로직만 공동 수행하는 대부분의 spring bean들과 데이터베이스 연결 정보를 가진 DataSource 등 공통 기술 지원 객체 등에 사용
    - 프로토타입 스코드 (Prototype Scope)
        - bean 조회 요청이 발생할 때마다 컨테이너가 새로운 bean을 독립적으로 매번 생성하여 반환하는 특수 스코프 방식
        - bean에 `@Scope(ConfigurableBeanFactory.SCOPE_PROTOTYPE)` 어노테이션 추가
        - 스프링 컨테이너는 프로토타입 bean의 생성, 의존 관계 주입, 초기화 롤백까지만 관리하므로, bean이 사용한 자원의 반납(리소스 해제)은 해당 bean을 제공받은 클라이언트 코드가 직접 책임지고 처리해야 함
        - 생성과 관리의 복잡성 해소를 위해 bean으로 관리하는 것보다 new 연산자를 이용한 직접 생성이 권장됨

        ```java
        @Component
        @Scope(ConfigurableBeanFactory.SCOPE_PROTOTYPE) // "prototype"
        public class GasolineCar implements Car { }
        ```

- Spring Bean 생명주기 및 콜백
    - 생명주기 콜백의 필요성
        - 애플리케이션 시작(구동) 시점에 데이터베이스 커넥션 풀을 사전 연결해 두거나 외부 네트워크 소켓 세션을 연결하는 작업, 그리고 서버가 정지(종료)되기 직전에 열려 있던 커넥션을 부드럽게 닫고(Graceful Shutdown) 메모리 버퍼 자원을 안전하게 반환하는 등 외부 리소스를 연동 제어할 때 사용함
        - 생성자는 객체 내부의 구조를 채우고 메모리를 할당하는 최소한의 책임(생성 자체)에만 집중함
        - 따라서 데이터베이스 커넥션 풀 사전 연결이나 네트워크 소켓 연결 등의 무거운 외부 리소스 작업은 생성자에서 수행하지 않고, 객체 생성이 완전히 끝나고 의존성 주입이 보장된 이후 호출되는 초기화 콜백 단계에서 실행해야 함
        - 특히 필드 주입(Field Injection)이나 수정자 주입(Setter Injection)을 사용하는 경우, 생성자가 호출되는 시점에는 주입받을 bean들이 아직 null 상태이므로 생성자 내에서 의존 객체를 활용한 초기화 코드를 실행하면 NullPointerException 예외가 발생함. 따라서 이 주입 방식을 사용할 때는 의존 관계 연동을 위한 초기화 메소드의 활용이 필수적
    - Spring Bean 생명주기 전체 흐름
        1. 스프링 컨테이너 기동
        2. spring bean 인스턴스 생성
        3. 의존 관계 주입 완료
        4. 초기화 메서드 수행(@PostConstruct)
        5. bean의 기동 및 비즈니스 로직 수행
        6. 소멸 메서드 수행(@PreDestroy)
        7. 스프링 컨테이너 종료 및 리소스 소멸
    - Bean의 생명주기 콜백 지정 방식
        - 어노테이션 지정 방식
            - bean 내부의 실행 메소드 상단에 @PostConstruct(초기화 메소드) 및 @PreDestroy(소멸 메소드) 자바 표준 어노테이션을 지정
            - Jakarta Annotations API 라이브러리 필요
            - 자바 표준 어노테이션이므로 스프링에 종속되지 않아서 사용을 권장하는 방식
        - 인터페이스 상속 방식
            - InitializingBean(afterPropertiesSet 구현) 및 DisprosableBean(destroy 구현) 인터페이스 직접 구현을 통한 제어
            - 스프링 프레임워크 기술 규격에 코드가 종속되므로 사용을 원하지 않음
        - bean 설정 옵션 방식
            - 자바 bean 설정 작성 시 `@Bean(initMethod = “메소드명”, destroyMethod = “메소드명”)` 형태로 콜백 이름 기입 및 지정

        ```java
        import jakarta.annotation.PostConstruct;
        import jakarta.annotation.PreDestroy;
        import org.springframework.beans.factory.annotation.Value;
        import org.springframework.stereotype.Component;
        
        @Component
        public class TempFileSupport {
            @Value("resoueces/temp.log") // 단순한 값을 주입할 때 사용
            private String tempFilePath;
        
            public TempFileSupport() {
                System.out.println("생성자 호출: 객체가 생성됨. 임시파일경로 = " + tempFilePath);
            }
        
            // 1. 초기화 콜백 (의존 관계 주입이 끝난 직후 가동되어 임시 작업 파일 생성)
            @PostConstruct
            public void initTempFile() {
                System.out.println("초기화 콜백: 작업용 임시 파일 생성 완료. 경로 = " + tempFilePath);
            }
        
            // 2. 비즈니스 기능 실행
            public void writeLog(String logMessage) {
                System.out.println("임시 파일에 로그 기록: " + logMessage);
            }
        
            // 3. 소멸 콜백 (컨테이너 종료 직전에 가동되어 자원 해제 및 파일 자동 삭제)
            @PreDestroy
            public void clearTempFile() {
                System.out.println("소멸 콜백: 임시 파일 안전하게 삭제 및 디스크 자원 반환.");
            }
        }
        ```

    - 설정 클래스 등록 (AppConfig)

        ```java
        @Configuration
        @ComponentScan
        public class AppConfig { }
        ```

    - 메인 구동 및 로그 검증 (SpringCoreApplication)

        ```java
        public class SpringCoreApplication {
            public static void main(String[] args) {
                // 1. 컨테이너 기동 및 설정 정보 로딩 (초기화 콜백 자동 발동)
                AnnotationConfigApplicationContext context = 
                        new AnnotationConfigApplicationContext(AppConfig.class);
        
                // 2. 빈 조회 및 비즈니스 사용
                TempFileSupport support = context.getBean(TempFileSupport.class);
                support.writeLog("사용자 로그인 작업 수행 로그 기록.");
        
                // 3. 스프링 컨테이너 종료
                context.close();
            }
        }
        ```

    - 콘솔 출력 로그

        ```
        생성자 호출: 객체가 생성됨. 임시파일경로 = null
        초기화 콜백: 작업용 임시 파일 생성 완료. 경로 = resoueces/temp.log
        임시 파일에 로그 기록: 사용자 로그인 작업 수행 로그 기록.
        소멸 콜백: 임시 파일 안전하게 삭제 및 디스크 자원 반환.
        ```


## 5~6: 스프링 부트 개요

### 스프링부트 개요

- 스프링 부트 (Spring Boot)
    - 기존 스프링 프레임워크의 번거로운 환경 설정이나 객체를 복잡하게 등록하는 과정을 알아서 자동으로 해결해 주고 필요한 라이브러리들의 버전 충돌까지 스스로 관리해 주어, 결과적으로 스프링 개발을 훨씬 쉽고 편리하게 진행할 수 있도록 지원해 주는 프레임워크
    - 웹 서버(Tomcat 등)를 내장하고 있어 별도의 WAS(Web Application Server) 설치 없이 단독 실행 가능한 애플리케이션 빌드를 지원함.
- 스프링 프레임워크와 스프링 부트의 차이점
    - 라이브러리 버전 조율
        - 스프링 프레임워크
            - 연동하려는 라이브러리 간 버전 충돌을 방지하기 위해 개발자가 하나씩 버전을 지정하고 호환성을 직접 테스트해야 함.
        - 스프링 부트
            - 특정 개발 목적(웹 개발, 데이터베이스 연동, 보안 설정 등)에 필요한 관련 라이브러리 목록을 한 묶음으로 묶어낸 스타터(Starter) 의존성 패키지를 제공함.
            - 개발자가 웹 개발용 스타터(spring-boot-starter-web) 하나만 추가해 두면, 내부적으로 호환되는 Spring Web MVC, Tomcat WAS, Jackson JSON 변환기 등 관련 라이브러리가 검증된 최적 버전 세트로 한 번에 도입되어 의존성 충돌과 복잡한 세부 버전 설정을 방지함.
    - 웹 서버 설정 및 배포 방식
        - 스프링 프레임워크
            - Tomcat 등 외부 WAS(Web Application Server)를 별도 설치하고, 프로젝트를 WAR 파일로 빌드하여 WAS에 배포 및 실행해야 함.
        - 스프링 부트
            - 내장 Tomcat 서버를 탑재하여 단독 실행 가능한 JAR 파일로 컴파일 및 자동 빌드하여 실행
- 스프링 부트 스타터 라이브러리
    - 특정 목적(웹 개발, DB 연동 등)에 맞춰 상호 검증된 호환 라이브러리들을 한 묶음으로 패키징하여 제공하는 의존성 조합 규격
    - 스타터 하나만 가져오면 연관된 수십 개의 하부 의존 라이브러리와 적합한 버전 정보가 gradle에 의해 한 번에 자동 로딩되어 버전 관리 복잡성을 대폭 낮춤
    - 스타터 라이브러리의 네이밍 규칙
        - 스프링 공식 스타터: `spring-boot-starter-기술명` 형식으로 명명하며, 스프링 공식 개발 팀에서 품질을 직접 보증하고 묶어 제공하는 스타터 (예: `spring-boot-starter-web`, `spring-boot-starter-security`)
        - 서드파티 및 오픈소스 스타터: `기술명-spring-boot-starter` 형식으로 명명하며, 외부 벤더사나 커뮤니티 진영에서 자사의 모듈을 스프링 부트에 맞게 조립하여 개별 제공하는 스타터 (예: `mybatis-spring-boot-starter`)
- 대표적인 라이브러리와 스타터
    - Spring Boot Starter (`spring-boot-starter`)
        - 프로젝트 생성 시 기본 탑재되는 최상위 스타터로 스프링 코어 컨텍스트, DI 컨테이너, 로깅 모듈 등을 내장함.
    - Spring Web (`spring-boot-starter-web`)
        - 웹 애플리케이션 개발을 지원하는 스타터 (Tomcat 서버 내장, 웹 개발에 필요한 라이브러리 포함).
    - Lombok (`org.projectlombok:lombok`)
        - getter, setter 등 반복적인 표준 코드 자동 완성 지원.
    - Spring Data JDBC (`spring-boot-starter-data-jdbc`)
        - 가벼운 데이터베이스 연동 지원 (단순 SQL 매핑 방식).
    - H2 Database (`com.h2database:h2`)
        - 개발 및 테스트용 인메모리 경량 관계형 데이터베이스 지원.
    - Spring Data JPA (`spring-boot-starter-data-jpa`)
        - ORM 기술 표준인 JPA를 활용한 고도화된 데이터베이스 영속성 제어 지원.
    - MySQL Driver (`mysql-connector-j`)
        - 상용 서비스 개발 시 주로 사용하는 MySQL 관계형 데이터베이스 연동 드라이버 제공.
    - Thymeleaf (`spring-boot-starter-thymeleaf`)
        - 서버 사이드 HTML 뷰 템플릿 동적 렌더링 지원.
    - MyBatis Framework (`mybatis-spring-boot-starter`)
        - 객체와 SQL 간 매핑을 통해 편리한 데이터베이스 영속성 제어 지원.
    - Spring Security (`spring-boot-starter-security`)
        - 세션 기반 사용자 인증 및 권한 부여 등 웹 보안 설정 제어.

### 스프링 부트 빌드 설정

- build.gradle 설정 분석
    - build.gradle

    ```java
    plugins {
        id 'java'
        id 'org.springframework.boot' version '4.1.0'
        id 'io.spring.dependency-management' version '1.1.7'
    }
    
    group = 'net.likelion.bebc25'
    version = '0.0.1-SNAPSHOT'
    
    java {
        toolchain {
            languageVersion = JavaLanguageVersion.of(25)
        }
    }
    
    repositories {
        mavenCentral()
    }
    
    dependencies {
        implementation 'org.springframework.boot:spring-boot-starter'
        implementation 'org.springframework.boot:spring-boot-starter-web'
        compileOnly 'org.projectlombok:lombok'
        runtimeOnly 'com.mysql:mysql-connector-j'
        annotationProcessor 'org.projectlombok:lombok'
        testImplementation 'org.springframework.boot:spring-boot-starter-test'
        testRuntimeOnly 'org.junit.platform:junit-platform-launcher'
    }
    
    tasks.named('test') {
        useJUnitPlatform()
    }
    ```

    - 플러그인 설정 (plugins)
        - 프로젝트 빌드 프로세스에 필요한 추가 기능과 컴파일/실행 도구를 그레이들에 적용하는 설정 영역.

        ```java
        plugins {
          id 'java'
        }
        ```

        - java: 그레이들이 자바 프로젝트를 인식하여 컴파일 및 빌드할 수 있는 표준 자바 기능 적용.
    - 프로젝트 식별 설정 (group, version)
        - 빌드된 산출물(JAR 등)을 배포하거나 공유할 때 사용하는 라이브러리의 고유 식별 정보.

        ```java
        group = 'net.likelion.bebc25'
        version = '0.0.1-SNAPSHOT'
        ```

        - group: 프로젝트 배포 시 다른 라이브러리와 식별하기 위해 단체의 도메인 기반 고유 그룹명 지정.
        - version: 애플리케이션의 개발 버전을 명시하며, 개발 중인 단계는 주로 SNAPSHOT 명칭 부여.
    - 자바 툴체인 설정 (java.toolchain)
        - 로컬 PC의 환경 변수(JAVA_HOME)와 상관없이 빌드와 컴파일 및 실행에 사용할 JDK 버전을 일관되게 강제하는 기능.

        ```java
        java {
          toolchain {
            languageVersion = JavaLanguageVersion.of(25)
          }
        }
        ```

        - languageVersion: 프로젝트가 동작할 JVM 런타임을 JDK 25 버전으로 강제하며, 만약 현재 PC에 없으면 gradle이 원격 저장소에서 해당 버전을 자동으로 다운로드하여 빌드에 영구적으로 캐싱하여 사용.
    - 라이브러리 저장소 설정 (repositories)
        - 프로젝트가 사용하는 외부 의존 라이브러리 파일들을 네트워크를 통해 받아올 온라인 서버 저장소 선언.

        ```java
        repositories {
          mavenCentral()
        }
        ```

        - mavenCentral(): 메이븐 중앙 저장소를 라이브러리 다운로드 경로로 지정.
    - 의존성 설정 키워드 (dependencies)
        - 컴파일, 테스트, 실행 시 활용할 라이브러리들을 그레이들이 다운로드하도록 지정.

        ```java
        dependencies {
            implementation 'org.springframework.boot:spring-boot-starter'
            implementation 'org.springframework.boot:spring-boot-starter-web'
            compileOnly 'org.projectlombok:lombok'
            runtimeOnly 'com.mysql:mysql-connector-j'
            annotationProcessor 'org.projectlombok:lombok'
            testImplementation 'org.springframework.boot:spring-boot-starter-test'
            testRuntimeOnly 'org.junit.platform:junit-platform-launcher'
        }
        ```

        - 메인 소스코드 영역에 적용되는 속성 (src/main)
            - compileOnly: 컴파일에만 적용. 자바 소스 컴파일 시점에만 사용되고, 실제 프로그램 실행(런타임) 단계에는 제외되므로 최종 실행 파일(JAR)의 용량을 줄이는 설정.
                - lombok: 코드 내 반복해서 적어야 하는 표준 메서드(getter, setter 등) 작성을 어노테이션만으로 자동 완성하는 개발 생산성 도구.
            - implementation: 컴파일과 런타임 모두 적용. 메인 소스 영역의 컴파일과 실제 실행(런타임) 모두에 공통 사용되며, 테스트 소스 영역으로 자동 상속되어 테스트 수행 시에도 사용되는 일반적인 라이브러리 지정.
                - spring-boot-starter: 프로젝트 생성 시 기본 탑재되는 최상위 스타터로 스프링 코어 컨텍스트, DI 컨테이너, 로깅 모듈 등을 내장함.
                - spring-boot-starter-web: 내장 톰캣 서버를 포함하여 Spring MVC 기반의 웹 애플리케이션 개발 환경을 자동으로 구축함.
            - annotationProcessor: 특수 컴파일 설정. 컴파일 시 자바 어노테이션 정보들을 스캔하여 실질적인 클래스 및 바이트코드로 자동 조립하고 변환해 주는 전용 컴파일 엔진.
                - lombok: 롬복 어노테이션을 분석하여 자바 클래스 파일 내에 코드를 끼워 넣고 빌드해 주는 컴파일용 플러그인.
            - runtimeOnly: 런타임에만 적용. 소스코드 컴파일 시점에는 사용되지 않으나, 실제 애플리케이션 작동(런타임) 단계에 로드되어 구동을 보조하는 라이브러리 지정.
                - mysql-connector-j: 자바 프로그램과 MySQL 데이터베이스를 통신 인터페이스로 연결해 주는 JDBC 드라이버 라이브러리.
        - 테스트 소스코드 영역에 적용되는 속성 (src/test)
            - testRuntimeOnly: 런타임에만 적용. 테스트 코드를 작성하고 컴파일할 때는 쓰이지 않고, 테스트가 실제로 작동하는 테스트 런타임 단계에만 사용되어 구동을 보조하는 엔진 설정.
                - junit-platform-launcher: intellij 등의 개발 도구나 gradle이 작성된 JUnit 5 테스트를 정상적으로 인식하여 구동하고 그 결과를 수집할 수 있게 돕는 테스트 런처 엔진.
            - testImplementation: 컴파일과 런타임 모두 적용. 테스트 소스 영역의 컴파일과 실행 모두에 공통 사용되며, 실제 배포용 메인 소스 영역에는 포함되지 않도록 완벽하게 차단되는 검증 라이브러리 지정.
                - spring-boot-starter-test: JUnit 5, Mockito, AssertJ 등 스프링 부트 애플리케이션 테스트에 필요한 검증 라이브러리를 내장함.
    - 태스크 제어 설정 (tasks.named)
        - gradle이 기본 제공하는 작업을 가로채어 세부적인 작업 속성을 변경하는 설정.

        ```java
        tasks.named('test') {
          useJUnitPlatform()
        }
        ```

        - useJUnitPlatform(): 테스트 실행 태스크를 타깃으로 삼아, 자바의 표준 테스트 엔진인 JUnit 5를 활용해 작동하도록 지정

### 스프링 부트의 자동 빈 등록 메커니즘

- 메인 실행 클래스와 @SpringBootApplication
    - 스프링 부트 애플리케이션의 시작점 Application 클래스 위에 @SpringBootApplication 어노테이션을 부여하여 기동함
    - 이 단일 어노테이션은 스프링 부트 자동화의 핵심인 다음 3대 어노테이션을 포함하고 있음
        - @SpringBootConfiguration: 스프링 부트 전용 설정 클래스임을 명시하며 내부적으로 @Configuration 기능을 공유함
        - @EnableAutoConfiguration: 사전에 준비된 스프링의 기성 컴포넌트 설정을 클래스패스 라이브러리 탑재 여부에 따라 컨테이너에 자동 등록해 줌
        - @ComponentScan: 컴포넌트 탐색을 수행함. 이 설정 클래스가 위치한 패키지를 루트 패키지로 삼아 하위 모든 패키지의 @Component 계열 빈들을 스캔함
- 컴포넌트 스캔의 탐색 경로 일탈 문제
    - 기동 클래스가 `net.likelion.bebc25.initializr` 패키지에 위치한 상황에서, 개발자가 패키지 경로를 벗어난 형제 패키지(예: `net.likelion.bebc25.external`)에 @Component bean을 작성하면 스프링이 이 컴포넌트를 스캔하지 못해 자동 주입 단계에서 NoSuchBeanDefinitionException 예외를 발생시킴
    - 이를 해결하기 위해서는 @SpringBootApplication 어노테이션 내부에 scanBasePackages 속성을 명시하여 탐색 범위를 넓혀주거나 루트 패키지로 이동시켜 결합해야 함

    ```java
    @SpringBootApplication(scanBasePackages = "net.likelion.bebc25")
    public class SpringBootStarterInitializrApplication {
        public static void main(String[] args) {
            SpringApplication.run(SpringBootStarterInitializrApplication.class, args);
        }
    }
    ```

- 자동 의존성 주입 예시

    ```java
    package net.likelion.bebc25.initializr;
    
    public interface Car {
        void startEngine();
        void drive();
        void stopEngine();
    }
    ```

    ```java
    package net.likelion.bebc25.initializr;
    
    import org.springframework.stereotype.Component;
    
    @Component
    public class GasolineCar implements Car {
        public void startEngine() {
            System.out.println("가솔린 엔진에 점화하여 시동을 켭니다.");
        }
        public void drive() {
            System.out.println("가솔린 동력으로 주행합니다.");
        }
        public void stopEngine() {
            System.out.println("가솔린 공급을 차단하여 시동을 끕니다.");
        }
    }
    ```

    ```java
    package net.likelion.bebc25.initializr;
    
    import org.springframework.stereotype.Component;
    
    @Component
    public class Driver {
        private final Car car;
    
        // 단일 생성자이므로 @Autowired 어노테이션 생략을 통한 의존성 주입 완료
        public Driver(Car car) {
            this.car = car;
        }
    
        public void driveCar() {
            car.startEngine();
            car.drive();
            car.stopEngine();
        }
    }
    ```


### 스프링 부트 설정 파일과 프로파일

- Properties와 YAML 설정 포맷 비교
    - 설정을 정의할 때 사용하는 application.properties와 application.yml의 문법 및 구조적 특징을 비교함
    - Properties 설정 (`application.properties`)

    ```
    server.port=8080
    spring.application.name=spring-boot-intellij
    ```

    - YAML 설정 (`application.yml`)

    ```
    server:
      port: 8080
    spring:
      application:
        name: spring-boot-intellij
    ```

  | **구분** | **Properties** | **YAML** |
      | --- | --- | --- |
  | **구조** | 평면적인 key=value 구조 | 들여쓰기 공백 기준의 계층형 트리 구조 |
  | **장점** | 작성법이 단순하여 들여쓰기 실수로 인한 에러가 없음 | 접두사의 중복 작성을 배제하여 구조가 간결하고 가독성이 뛰어남 |
  | **단점** | 중복되는 설정 키명을 매 줄마다 입력해야 해서 비대해짐 | 공백 1칸의 정렬 오타에도 로드 실패 예외를 유발함 |
- 프로파일(Profile)을 활용한 배포 환경 격리
    - 프로파일은 구동 환경(로컬 개발, 스테이징, 실운영 환경)에 따라 데이터베이스 연결 주소나 포트 등 완전히 격리된 설정을 적용하기 위한 환경 세트 분류 장치
    - 설정 파일명에 `application-{profile}.yml` 형식의 작명 관례를 도입하여 각 환경 설정을 분리함
    - 활성화 지정 방법:
        - 기본 application.yml에 spring.profiles.active 속성을 부여하여 가동 환경을 결정함
        - 빌드 배포 단계에서는 자바 커맨드 컴파일 파라미터(`Dspring.profiles.active=prod`) 지정을 통해 동적으로 변경함

    ```java
    spring:
      profiles:
        active: local
    ```

- @Profile 어노테이션을 활용한 조건부 Bean 등록
    - 특정 프로파일이 활성화된 환경에서만 스프링 컨테이너에 등록되는 특수 bean을 정의할 때 사용함
    - 실무에서는 로컬 개발 단계의 모조(Mock) 결제 수단 bean과 실서버에서 가동할 실제 카드 결제 API bean을 안전하게 격리 등록할 때 활용함

    ```java
    package net.likelion.bebc25.initializr;
    
    public interface PaymentService {
        void processPayment(int amount);
    }
    ```

    ```java
    package net.likelion.bebc25.initializr;
    
    import org.springframework.context.annotation.Profile;
    import org.springframework.stereotype.Component;
    ```

### 로깅(Logging) 시스템

- System.out.println의 한계와 로깅 프레임워크
    - System.out.println을 사용하면 콘솔에 무조건 텍스트가 표시되므로 로그 레벨 통제가 전혀 불가능하고, 동기적 입출력 처리로 인해 애플리케이션의 연산 처리 속도를 심각하게 떨어트려 실무 운영 서버 배포 시 사용이 전면 금지됨
    - 로깅 프레임워크는 출력 레벨 통제, 파일 및 원격 로깅 저장, 성능 최적화 등을 지원함
    - 스프링 부트는 로깅 명세 파사드(Facade)인 SLF4J를 기본 지원하고, 실제 내부 구현체로 Logback 라이브러리를 바인딩하여 기동함
- 롬복(Lombok) @Slf4j를 활용한 로그 기록 실습
    - 롬복의 @Slf4j 어노테이션을 클래스 상단에 붙여두면 자동으로 static final Logger 변수가 컴파일 단계에서 생성되어, 단순 호출만으로 정교한 로그를 기록할 수 있음

    ```java
    package net.likelion.bebc25.initializr;
    
    import jakarta.annotation.PostConstruct;
    import jakarta.annotation.PreDestroy;
    import java.io.FileOutputStream;
    import java.io.IOException;
    import lombok.extern.slf4j.Slf4j;
    import org.springframework.beans.factory.annotation.Value;
    import org.springframework.stereotype.Component;
    
    @Slf4j // 롬복 로깅 어노테이션 추가
    @Component
    public class TempFileSupport {
        @Value("resources/temp.log")
        private String tempFilePath;
        private FileOutputStream fos;
    
        public TempFileSupport() {
            log.info("생성자 호출: 객체가 생성됨. 임시파일경로 = {}", tempFilePath);
        }
    
        @PostConstruct
        public void initTempFile() {
            try {
                this.fos = new FileOutputStream(tempFilePath, true);
                log.info("초기화 콜백: 파일 출력 스트림 개방 완료. 경로 = {}", tempFilePath);
            } catch (IOException e) {
                log.error("스트림 개방 도중 파일 에러 발생", e);
            }
        }
    
        public void writeLog(String logMessage) {
            try {
                fos.write((logMessage + "\n").getBytes());
                log.info("임시 파일에 로그 기록: {}", logMessage);
            } catch (IOException e) {
                log.error("로그 데이터 작성 오류 발생", e);
            }
        }
    
        @PreDestroy
        public void clearTempFile() {
            try {
                if (fos != null) {
                    fos.close();
                    log.info("소멸 콜백: 파일 출력 스트림 안전하게 닫고 자원 반납 완료.");
                }
            } catch (IOException e) {
                log.error("스트림 자원 해제 오류 발생", e);
            }
        }
    }
    ```

- 패키지별 로그 레벨 제어 설정
    - 로그 레벨은 중요도에 따라 TRACE - DEBUG - INFO - WARN - ERROR 순으로 높아지며 기본 동작 값은 INFO로 지정됨
    - application.yml에서 특정 패키지를 지정하여 해당 패키지의 로그 필터 레벨을 DEBUG나 TRACE로 낮춤으로써 디버깅 상태 로그를 상세히 확보함

    ```java
    logging:
      level:
        root: info
        net.likelion.bebc25: debug // 지정한 패키지 하위의 로그는 DEBUG 단계까지 모두 콘솔에 노출함
    ```
